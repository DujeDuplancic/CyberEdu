================================================================================
1. PREGLED PROJEKTA
================================================================================

CyberEdu je web platforma za učenje cyber sigurnosti namijenjena studentima
i entuzijastima. Glavna ideja je spojiti TEORIJU (lekcije, wiki članci) s
PRAKSOM (CTF izazovi koje korisnik mora "hakirati" i pronaći flag).

KLJUČNE FUNKCIONALNOSTI:
- Registracija i prijava korisnika s admin/regular ulogama
- CTF (Capture The Flag) izazovi raspoređeni po kategorijama
  (Reverse Engineering, Binary Exploitation, Cryptography, Web Security,
   Steganography, itd.)
- Sustav bodova, ranking i leaderboard
- Achievementi (postignuća) koja se automatski otključavaju
- Wiki - baza znanja s člancima po kategorijama
- Video lekcije s YouTube embedom
- Community - forum tipa diskusije + odgovori
- Chat - poruke između korisnika u stvarnom vremenu (polling)
- AI Asistent SentinelAI - integracija s Google Gemini 2.5 Flash
- News - agregator vijesti iz svijeta cyber sigurnosti (RSS)
- Admin panel - kompletna kontrola nad svim sadržajem

CILJ:
Pružiti studentima sigurno okruženje za etičko hakiranje. Svi izazovi
imaju "flag" (npr. CTF{moja_zastavica}) koji korisnik mora otkriti
hakiranjem - bilo cracking, reverse engineering, web exploit, itd.

================================================================================
3. ARHITEKTURA - KAKO FRONTEND I BACKEND KOMUNICIRAJU
================================================================================

Frontend i backend su POTPUNO ODVOJENI. To se zove "decoupled architecture".

KORIŠTENJE:
1. Korisnik otvara http://localhost:5173 (React aplikacija)
2. React aplikacija šalje fetch() zahtjev na
   http://localhost/CyberEdu/Backend/[endpoint].php
3. PHP backend čita podatke iz MySQL-a i vraća JSON
4. React parsira JSON i prikazuje rezultat

CORS:
Pošto frontend i backend rade na različitim portovima (5173 vs 80),
svaki PHP endpoint mora imati CORS headere na vrhu:

   header("Access-Control-Allow-Origin: http://localhost:5173");
   header("Access-Control-Allow-Methods: GET, POST, OPTIONS");
   header("Access-Control-Allow-Headers: Content-Type, Authorization");
   header("Access-Control-Allow-Credentials: true");

OPTIONS preflight zahtjev se mora obraditi posebno:
   if ($_SERVER['REQUEST_METHOD'] == 'OPTIONS') {
       http_response_code(200);
       exit();
   }

AUTENTIFIKACIJA:
Koristi se localStorage na frontendu. Pri prijavi se cijeli user objekt
sprema u localStorage.user (kao JSON). Pri svakom pozivu prema backendu,
frontend šalje user.id u tijelu zahtjeva ili query string-u.

   localStorage.getItem('user')  ->  { id, username, email, is_admin, ... }

5. BAZA PODATAKA - TABLICE I VEZE
================================================================================

Baza podataka se zove "cyberedu". Sve tablice koriste InnoDB engine i UTF-8.

TABLICA: users (001_create_users_table.php)
-------------------------------------------
Glavna tablica korisnika.
  - id                INT PRIMARY KEY AUTO_INCREMENT
  - username          VARCHAR(50) UNIQUE NOT NULL
  - email             VARCHAR(100) UNIQUE NOT NULL
  - password_hash     VARCHAR(255) NOT NULL  (bcrypt hash, NIKAD plain text)
  - avatar_url        VARCHAR(255)            (npr. "uploads/avatars/xyz.jpg")
  - points            INT DEFAULT 0           (ukupni bodovi)
  - rank              INT DEFAULT 0
  - created_at        TIMESTAMP
  - is_admin          BOOLEAN DEFAULT FALSE

TABLICA: categories (002_create_categories_table.php)
-----------------------------------------------------
Kategorije izazova (Reverse Engineering, Web Security, itd.)
  - id, name UNIQUE, description, icon_name

TABLICA: challenges (003_create_challenges_table.php)
-----------------------------------------------------
CTF izazovi.
  - id, title, description, points, difficulty (Easy/Medium/Hard)
  - category_id        FK -> categories
  - flag               VARCHAR  (BCRYPT HASH, ne plain text!)
  - file_url           VARCHAR  (dodato u migraciji 013)
  - is_active          BOOLEAN
  - created_by         FK -> users
  - created_at         TIMESTAMP

VAŽNO: Flag se sprema kao bcrypt hash (password_hash funkcija). Pri submit-u
se koristi password_verify() - tako da NITKO ne može pročitati flag iz baze.

TABLICA: solves (004_create_solves_table.php)
---------------------------------------------
Tko je riješio koji izazov.
  - id, user_id (FK), challenge_id (FK), solved_at TIMESTAMP
  - UNIQUE KEY (user_id, challenge_id) - jedan user ne može riješiti isti izazov dvaput

TABLICA: lectures (005_create_lectures_table.php)
-------------------------------------------------
Video lekcije (YouTube embed-ovi).
  - id, title, description, video_url, thumbnail_url
  - duration_minutes, level (Beginner/Intermediate/Advanced)
  - instructor, category_id (FK), is_published, views

TABLICA: discussions (006_create_discussions_table.php)
-------------------------------------------------------
Forum teme.
  - id, title, content, author_id (FK), category, is_pinned, views, created_at

TABLICA: replies (007_create_replies_table.php)
-----------------------------------------------
Odgovori na forum teme.
  - id, discussion_id (FK), author_id (FK), content, created_at

TABLICA: achievements (008_create_achievements_table.php)
---------------------------------------------------------
Postignuća koja korisnik može osvojiti.
  - id, name, description, icon_url, points_reward
  - criteria_type ENUM('solves_count', 'points_total', 'category_master', 'streak')
  - criteria_value INT  (npr. 10 = "riješi 10 izazova")
  - category_id (FK, opcionalno - za category_master)
  - is_hidden BOOLEAN

TABLICA: user_achievements (009_create_user_achievements_table.php)
-------------------------------------------------------------------
Tko je dobio koje achievement.
  - user_id (FK), achievement_id (FK), unlocked_at

TABLICA: wiki_categories (014_create_wiki_categories_table.php)
TABLICA: wiki_articles (010_create_wiki_articles_table.php)
-----------------------------------------------------------
Wiki članci po kategorijama.
  - id, title, slug, content, excerpt, category_id (FK)
  - reading_time, difficulty_level, is_published, views

TABLICA: leaderboard_snapshots / leaderboard_entries (011, 012)
---------------------------------------------------------------
Cached snapshot-ovi leaderboard-a (za optimizaciju).

TABLICA: chat_messages (017_create_chat_messages_table.php) - DODANO U PROJEKTU
------------------------------------------------------------------------------
Privatne poruke između korisnika.
  - id, sender_id (FK), recipient_id (FK)
  - content TEXT
  - attachment_url, attachment_name, attachment_type (image/file)
  - is_read BOOLEAN
  - created_at TIMESTAMP
  - INDEX (sender_id, recipient_id, created_at) - za brže pretrage razgovora
  - INDEX (recipient_id, is_read) - za brži unread count


MIGRACIJSKI SUSTAV:
-------------------
Migracije se izvršavaju preko Backend/Migrations/force_migrate.php skripte.
Skripta:
1. Kreira tablicu 'migrations' ako ne postoji (za praćenje koje su pokrenute).
2. Učita sve fajlove iz mape /migrations/ sortirane po imenu (001, 002, ...).
3. Za svaki fajl provjeri je li već izvršen.
4. Ako nije, izvrši SQL i upiše u 'migrations' tablicu.

Tako se može sigurno ponovno pokretati - neće dva puta kreirati istu tablicu.

6. BACKEND - PHP ENDPOINT-i PO GRUPAMA
================================================================================

OPĆI PATTERN SVAKOG ENDPOINT-a:
-------------------------------
1. CORS headeri na vrhu
2. OPTIONS preflight handling
3. require_once '../config/database.php'
4. try/catch oko logike
5. echo json_encode([...]) na kraju

------ AUTH (Backend/auth/) ------

login.php
  POST { email, password }
  - Pronalazi usera po email-u
  - password_verify() s bcrypt hash-om
  - Vraća { success, user: { id, username, email, points, rank, is_admin } }
  Frontend onda taj user objekt sprema u localStorage.

register.php
  POST { username, email, password }
  - Validira input (email format, lozinka min duljina)
  - password_hash() PRIJE spremanja
  - INSERT INTO users
  - Vraća user objekt isto kao login.

------ ADMIN (Backend/admin/) ------

check_admin.php
  POST { user_id }
  - Vraća { success, is_admin } - frontend ovaj endpoint poziva pri ulasku
    na /admin rutu i preusmjerava ako user nije admin.

users.php
  GET     - lista svih korisnika
  POST    - { action: 'toggle_admin', user_id, admin_status } - promijeni admin status
          - { action: 'delete_user', user_id } - obriši korisnika
  Multi-action endpoint - dispatch po 'action' polju.

create_challenge.php / update_challenge.php / delete_challenge.php
  POST za svaki - obrada CRUD operacija nad izazovima.
  VAŽNO: flag se HASH-a (password_hash) prije INSERT-a.
  create_challenge sada vraća i 'challenge_id' kako bi frontend mogao
  uploadati file odmah nakon kreiranja.

create_lecture.php / update_lecture.php / delete_lecture.php
  Slično kao za izazove, ali za video lekcije.

create_wiki_article.php / update_wiki_article.php / delete_wiki_article.php
  Slično za wiki članke.

create_achievement.php / update_achievement.php / delete_achievement.php /
toggle_achievement_visibility.php
  Slično za postignuća.

stats.php
  GET - vraća { total_users, total_challenges, total_lectures, wiki_articles, total_achievements }
  Za admin dashboard kartice.

------ CHALLENGES (Backend/challenges/) ------

get_challenges.php
  GET ?user_id=X (opcionalno)
  - Vraća sve izazove s flag-om "solved" ako je proslijeđen user_id.
  - Bodovi se prikazuju, ali flag NIKAD nije u JSON-u.

submit_flag.php  <-- NAJVAŽNIJI ENDPOINT ZA CTF
  POST { user_id, challenge_id, flag }
  - Dohvaća izazov iz baze
  - password_verify($posted_flag, $stored_hashed_flag)
  - Ako se podudara:
    * INSERT INTO solves (user_id, challenge_id) - UNIQUE constraint sprečava duple
    * UPDATE users SET points = points + challenge.points
    * Provjeri može li user dobiti neki achievement (logika u submit_flag.php
      ili u functions.php)
  - Vraća { success, points } ili { success: false }

get_categories.php
  GET - lista kategorija (za filter chips).

get_daily_goal.php  <-- DODANO U PROJEKTU
  GET ?user_id=X
  - Konstanta DAILY_GOAL_TARGET = 3 (može se mijenjati).
  - Broji koliko je izazova rješeno DANAS (DATE(solved_at) = CURDATE()).
  - Računa streak (uzastopne dane s barem jednim solve-om).
  - Vraća { goal, solves_today, remaining, progress_pct, completed, streak_days }
  - Frontend koristi za "Daily Goal" widget na CTF stranici.

download.php
  GET ?file=... - sigurno serviranje challenge handout fajlova.

------ PROFILE (Backend/profile/) ------

get_profile.php
  GET ?user_id=X
  - Vraća user objekt + kategoriju progress + zadnja rješenja.
  - Računa rank dinamički: COUNT(*) + 1 gdje points > moji.
  - UKLJUČUJE id i email (popravljeno tijekom projekta).

update_profile.php  <-- POPRAVLJEN ZA PARCIJALNI UPDATE
  POST multipart/form-data ili JSON
  Fields (svi opcionalni):
    - user_id (obavezno)
    - username (ako prosljeđujemo - mijenjamo)
    - email
    - current_password + new_password (mijenja lozinku)
    - avatar (file)
    - remove_avatar=1 (briše postojeću sliku)
  - UPDATE upit se gradi DINAMIČKI iz $sets[] niza - polja koja nisu poslana
    nikad se ne dotaknu.

------ CHAT (Backend/chat/) - DODANO U PROJEKTU ------

get_contacts.php
  GET ?user_id=X
  - Vraća sve korisnike osim trenutnog.

8. KAKO FUNKCIONIRA - GLAVNI TIJEKOVI
================================================================================

------ TIJEK PRIJAVE ------

1. Korisnik upiše email + password u Login.jsx
2. Frontend → POST /auth/login.php
3. PHP:
   - SELECT * FROM users WHERE email = :email
   - password_verify($posted, $stored_hash)
   - Vrati cijeli user objekt
4. Frontend:
   - localStorage.setItem('user', JSON.stringify(data.user))
   - navigate('/')

Tako se user "logira" - zapamtimo ga u localStorage-u. Sve buduće provjere
gledaju localStorage. Pri logout-u: localStorage.removeItem('user').

------ TIJEK SUBMIT FLAGA ------

1. Korisnik klikne na izazov u CTF listi
2. Otvori se ChallengeModal s briefing-om i input poljem
3. Korisnik upiše flag (npr. CTF{moja_zastavica})
4. Frontend → POST /challenges/submit_flag.php { user_id, challenge_id, flag }
5. PHP:
   - SELECT flag FROM challenges WHERE id = :challenge_id
   - password_verify($posted_flag, $stored_hash)
   - Ako se podudara:
     * INSERT INTO solves
     * UPDATE users SET points = points + :challenge_points
     * Provjeri može li dobiti achievement (npr. solves_count == 10)
     * Ako da: INSERT INTO user_achievements
   - Vrati { success, points }
6. Frontend:
   - Animacija uspjeha (zelena)
   - showAchievement notifikacija ako je novi achievement
   - Refresh CTF liste (da ovaj izazov bude označen kao riješen)

------ TIJEK AI ASISTENTA ------

1. Korisnik upiše pitanje u Assistant.jsx (npr. "Daj mi hint za SQL injection")
2. Frontend dodaje poruku u messagesList state
3. Frontend → POST /ai/ai_assistant.php { messages: [...puna_povijest] }
4. PHP:
   - Konstruira payload za Gemini API:
     * systemInstruction (stroge upute - samo cyber security, ne otkrij flag)
     * contents (mapirane poruke)
   - cURL POST na Gemini API endpoint s API ključem
   - Parsira response
5. Frontend:
   - Dodaje asistentov odgovor u messages
   - Renderira mini-markdown (bold, code blocks, listovi)

------ TIJEK CHAT-a ------

INICIJALNI LOAD:
1. Korisnik otvori /chat
2. Frontend čita currentUser iz localStorage
3. GET /chat/get_contacts.php?user_id=X
4. Prikaže listu kontakata u sidebaru
5. Default-no odabire prvog kontakta s razgovorom

KLIK NA KONTAKT:
1. setActiveId(contact.id)
2. GET /chat/get_messages.php?user_id=X&with=Y
3. Backend AUTOMATSKI markira primljene poruke kao pročitane
4. Frontend prikaže messages

SLANJE PORUKE:
1. Korisnik upiše tekst, klikne Send
2. Frontend → POST /chat/send_message.php { sender_id, recipient_id, content }
3. Backend INSERT-a u chat_messages, vrati upravo spremljenu poruku
4. Frontend doda poruku u state (optimistically)

REAL-TIME ASPEKT (preko polling-a):
- Svakih 8s: refetch get_contacts (osvježi listu + unread count-ove)
- Svakih 3s: refetch get_messages (osvježi razgovor s novim porukama)
- Nije pravi WebSocket, ali daje feel real-time chat-a.

------ TIJEK ACHIEVEMENT-A ------

Achievementi su automatski - korisnik ih ne traži, samo se otključavaju.

Logika:
- Pri svakom solve-u (submit_flag.php), backend provjerava SVE achievementa.
- Za svaki achievement provjerava criteria:
  * solves_count → COUNT(solves WHERE user_id = X) >= criteria_value
  * points_total → users.points >= criteria_value
  * category_master → COUNT(solves JOIN challenges WHERE category_id = X) >= criteria_value
  * streak → izračun uzastopnih dana s aktivnošću
- Ako kriterij ispunjen, INSERT INTO user_achievements (ako već nije).
- Frontend pri sljedećem GET-u dohvaća user_achievements i prikazuje notifikaciju.

ARHITEKTURA:
- Tablica chat_messages drži sve poruke (sender, recipient, content, is_read).
- Polling: frontend periodički refetcha podatke.
- Indeksi na (sender_id, recipient_id, created_at) i (recipient_id, is_read)

GET_CONTACTS SQL TRIK:
- Za svakog korisnika izvlačimo zadnju poruku razgovora.
- Koristimo korelirani subquery umjesto window funkcija (kompatibilnost s
  starijim MySQL-om na XAMPP-u).
- ORDER BY (last_message_at IS NULL), last_message_at DESC, username ASC
  - NULL postavlja kontakte bez razgovora na kraj.
  - Inače sortira po vremenu zadnje poruke (najnovije prvi).

UNREAD COUNT:
- Računa se kao COUNT(*) WHERE sender = other AND recipient = me AND is_read = 0.
- Pri otvaranju razgovora, get_messages.php automatski radi
  UPDATE is_read = 1 - tako da se badge resetira čim user pogleda razgovor.

ATTACHMENTI:
- Tri polja: attachment_url, attachment_name, attachment_type ('image' ili 'file').
- Frontend uploada file preko zasebnog endpoint-a, pa onda send_message.php
  s URL-om od uploadanog fajla.
